# Phase 22.1: Self-Contained Export Service - Research

**Researched:** 2026-02-11
**Domain:** Decoupling export service from frontend, standalone Verovio rendering page, shared animation/camera module extraction
**Confidence:** HIGH

## Summary

The export service currently depends on the Next.js frontend's `/render` route -- Puppeteer navigates to `http://localhost:3000/render`, which loads RenderApp.tsx (a React component wrapping RegularRenderer). This phase eliminates that dependency by creating a standalone vanilla JS rendering page served by the export service's own Fastify server and extracting the animation/camera logic into a shared module that both the frontend and standalone page can use.

The core technical challenge is replicating the exact visual output of RegularRenderer in a vanilla JS page. This includes: Verovio SVG rendering with correct options, camera (vertical scroll) with 200ms ease-out transition simulation, notehead scale/color animations with per-frame delta computation, score region cropping, border SVG generation, and background image display. All of this logic currently lives inside a 1037-line React component with tight coupling to React hooks (useState, useEffect, useCallback, useRef, useMemo) and Zustand stores.

**Primary recommendation:** Extract the pure-logic portions of RegularRenderer's setTimestamp (camera + noteheads) and interpolation into a shared TypeScript module with zero React/Zustand dependencies. Bundle the standalone page using esbuild (already a transitive dep via tsx). Serve the bundle + verovio-toolkit-wasm.js from Fastify using @fastify/static (already installed).

### Pattern 1: Shared Animation Module (No React)
**What:** Extract the pure computation logic from RegularRenderer's `setTimestamp` callback into a standalone module
**When to use:** Any code that both the React renderer and the standalone page need
**Key insight:** The setTimestamp function in RegularRenderer (lines 586-780) is already nearly framework-agnostic -- it operates on DOM elements via querySelector, uses refs only for state, and performs no React state updates. It can be extracted with minimal changes.

The shared module needs to encapsulate:
1. **Camera computation:** binary search for current event, clamp camera Y, 200ms ease-out transition simulation
2. **Notehead animation:** delta-based per-frame animation (scale + color), active window tracking, reset logic
3. **Color interpolation:** hex/rgb parsing and linear interpolation
4. **Cubic bezier easing:** cssEaseOut function for camera transition

### Pattern 2: Standalone Verovio Page (No Bundler)
**What:** A single HTML file that loads verovio-toolkit-wasm.js via script tag, reads __EXPORT_CONFIG__, renders the score, and exposes window.animationController
**When to use:** For the export service's /render route
**Key insight:** The verovio-toolkit-wasm.js file (6.5MB) is a self-contained UMD script that includes WASM and all fonts inline. It can be loaded via a `<script>` tag and provides `verovio.module` global. This avoids needing esbuild entirely for WASM handling.

**Recommended approach:** Use esbuild to bundle only the TypeScript animation/border logic into a single JS file. The HTML page loads verovio-toolkit-wasm.js via script tag and the bundled animation JS separately.

### Anti-Patterns to Avoid
- **Bundling verovio WASM with esbuild:** The WASM is embedded in the 6.5MB JS file. Trying to import verovio/wasm + verovio/esm through esbuild creates WASM loading complications. Use the pre-built verovio-toolkit-wasm.js instead.
- **Using React in the standalone page:** The entire point is independence from the frontend. No React, no Zustand.
- **Copy-pasting RegularRenderer logic:** Extract into a shared module. Don't maintain two copies.
- **Modifying the multipart upload endpoint:** The existing POST /api/export already handles multipart uploads correctly with files + JSON fields. It does NOT need changes.

## Common Pitfalls

### Pitfall 1: Verovio Toolkit API Differences
**What goes wrong:** The standalone page's verovio-toolkit-wasm.js exposes a different API surface than the ESM import used in the frontend
**Why it happens:** The UMD build (`verovio-toolkit-wasm.js`) attaches to `window.verovio` with `verovio.module` (Promise) and `new verovio.toolkit()`. The ESM build uses `import { VerovioToolkit } from 'verovio/esm'` + `import createVerovioModule from 'verovio/wasm'`.
**How to avoid:** Test the exact initialization sequence with the UMD build. The toolkit API (setOptions, loadData, renderToSVG, renderToTimemap, getPageWithElement) is identical once created -- only the creation path differs.
**Warning signs:** `verovio is not defined` or `verovio.module is not a function` errors in Puppeteer console

### Pitfall 2: CSS Transform Scale Factor in Position Computation
**What goes wrong:** Event Y positions are incorrect because the standalone page uses a CSS scale transform (just like RenderApp) but computeEventPositions divides by domScale
**Why it happens:** RenderApp applies `transform: scale(viewportWidth / 980)` to fit the 980px-wide score into the viewport. The domScale detection in computeEventPositions uses `getBoundingClientRect().width / clientWidth` to detect this.
**How to avoid:** The standalone page MUST apply the same CSS scale transform pattern. The domScale detection is already built into computeEventPositions and will work automatically if the DOM structure matches.
**Warning signs:** Camera scrolling is off -- either jumping too far or not far enough

### Pitfall 3: Race Condition: Config Injection vs Page Load
**What goes wrong:** The standalone page script runs before __EXPORT_CONFIG__ is available
**Why it happens:** evaluateOnNewDocument injects the config before navigation, but if the page script executes in a deferred/async way, timing can be tricky
**How to avoid:** The current pattern already handles this correctly -- evaluateOnNewDocument runs before ANY page script. The standalone page reads `window.__EXPORT_CONFIG__` synchronously at the top of its entry point. Keep this pattern.
**Warning signs:** Config is undefined, empty score

### Pitfall 4: Page Top Margin Trimming
**What goes wrong:** Pages 2+ have visible gaps between them, causing camera positioning to be off
**Why it happens:** Verovio adds internal padding above the first system on each page. The frontend's useVerovio hook calls trimPageTopMargin() for pages 2+ to adjust the viewBox.
**How to avoid:** The standalone page MUST replicate the trimPageTopMargin logic exactly -- regex-based viewBox and height adjustment.
**Warning signs:** Vertical gaps between pages, camera positions don't match frontend preview

### Pitfall 5: rendererReady Signal Timing
**What goes wrong:** Puppeteer's waitForFunction times out because rendererReady is never set
**Why it happens:** The standalone page must set `window.rendererReady = true` only AFTER: Verovio renders all pages, events are extracted, positions are computed, and animationController is exposed. If any step fails silently, the signal is never set.
**How to avoid:** Follow the exact sequence: init Verovio -> render pages -> extract timemap -> mount SVG in DOM -> compute positions -> expose animationController -> set rendererReady. Add console.log at each step for debugging.
**Warning signs:** Puppeteer timeout on "waiting for renderer to signal readiness"

### Pitfall 6: @fastify/static Double Registration
**What goes wrong:** Fastify throws "The decorator 'sendFile' has already been added" error
**Why it happens:** @fastify/static registers a `reply.sendFile()` decorator by default. Registering it twice causes a collision.
**How to avoid:** The existing registration already uses `decorateReply: false`. The new registration must also use `decorateReply: false`, OR use a different approach (e.g., register once with multiple roots, or use reply.sendFile from a route handler with an explicit path).
**Warning signs:** Server startup crash with decorator collision error

### Pitfall 7: Border Rendering Without React
**What goes wrong:** Borders look different or are missing in the export
**Why it happens:** The frontend borders are React components (JSX) that generate SVG elements. The standalone page needs to generate the same SVG as HTML strings.
**How to avoid:** Port the border functions to return SVG strings instead of React elements. The SVG content is simple (lines, paths, circles, ellipses) and translates directly to string concatenation.
**Warning signs:** Missing borders in exported video, different border positioning

### Shared Animation Module - setTimestamp Core Logic
```typescript
// Source: Extracted from src/renderers/RegularRenderer.tsx lines 586-780
// This is the pure-logic version without React hooks

function cssEaseOut(t: number): number {
  // cubic-bezier(0, 0, 0.58, 1)
  if (t <= 0) return 0;
  if (t >= 1) return 1;
  const cx = 0, bx = 3 * (0.58 - 0), ax = 1 - cx - bx;
  const cy = 0, by = 3 * (1 - 0), ay = 1 - cy - by;
  let s = t;
  for (let i = 0; i < 8; i++) {
    const xVal = ((ax * s + bx) * s + cx) * s - t;
    const dxVal = (3 * ax * s + 2 * bx) * s + cx;
    if (Math.abs(dxVal) < 1e-6) break;
    s -= xVal / dxVal;
  }
  s = Math.max(0, Math.min(1, s));
  return ((ay * s + by) * s + cy) * s;
}

export function setTimestamp(
  seconds: number,
  events: { computedTimestamp: number; y: number; svgIds: string[] }[],
  state: AnimationState,
  config: AnimationConfig,
  cameraEl: HTMLElement,
  scoreEl: HTMLElement,
): void {
  // Binary search for current event
  let low = 0, high = events.length - 1, currentIndex = -1;
  while (low <= high) {
    const mid = Math.floor((low + high) / 2);
    if (events[mid].computedTimestamp <= seconds) {
      currentIndex = mid; low = mid + 1;
    } else { high = mid - 1; }
  }
  if (currentIndex < 0) return;

  // Camera: simulate 200ms ease-out transition
  const eventY = events[currentIndex].y;
  const viewportHeight = config.scoreRegionHeight ?? config.containerHeight;
  let newTarget = Math.max(0, Math.min(
    eventY - viewportHeight / 2,
    Math.max(0, config.totalHeight - viewportHeight)
  ));

  if (Math.abs(newTarget - state.transitionTarget) > 0.5) {
    state.transitionFrom = state.cameraY;
    state.transitionTarget = newTarget;
    state.transitionStart = seconds;
  }

  const elapsed = seconds - state.transitionStart;
  state.cameraY = elapsed >= 0 && elapsed < 0.2
    ? state.transitionFrom + (state.transitionTarget - state.transitionFrom) * cssEaseOut(elapsed / 0.2)
    : state.transitionTarget;

  cameraEl.style.transform = `translateY(${-state.cameraY}px)`;

  // Notehead animation (delta-based) -- same logic as RegularRenderer
  // [notehead animation code follows same pattern]
}
```

async function initVerovio(config) {

  const EDITOR_WIDTH = 980;
