# Phase 22.1: Self-Contained Export Service - Context

**Gathered:** 2026-02-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Make the export service fully self-contained — it bundles its own standalone rendering page, serves it from its own Express server, and Puppeteer captures frames without depending on the frontend. The frontend's /render route, RenderApp.tsx, and related export-mode code are removed. The export UI (button, progress) in the frontend stays.

</domain>

<decisions>
## Implementation Decisions

### Data transfer
- Frontend sends score file, audio file, settings JSON, and background image (if set) as a single multipart API request to a new endpoint (e.g., POST /api/export)
- Settings include all project data: colors, scale, font, border, animation options, score region, sync anchors
- Background image is included in the upload if one is set
- Old export trigger endpoint is removed, replaced by the new multipart endpoint

### Rendering approach
- Export service has its own standalone HTML page (vanilla JS, no React) that uses Verovio directly
- Core animation/camera logic is extracted into a shared module (no React dependency) that both the frontend React components and the export service's standalone page import
- Export service's Express server serves the standalone page at its own /render route (e.g., localhost:3001/render)
- Puppeteer navigates to the export service's own /render, not the frontend
- Puppeteer continues to use window.__EXPORT_CONFIG__ via evaluateOnNewDocument to pass data to the standalone page

### Frontend cleanup
- Delete the frontend's /render route entirely (src/app/render/page.tsx + client.tsx)
- Delete RenderApp.tsx from the frontend
- Remove /render from proxy.ts PUBLIC_PATHS since the route no longer exists
- Keep the export UI in App.tsx (Export Video button, WebSocket progress) — it now sends the multipart request instead of just triggering an export

### Claude's Discretion
- WebSocket progress reporting mechanism (keep existing or adjust)
- Exact structure of the shared animation/camera module
- How uploaded files are stored temporarily on the export service
- Build tooling for the standalone HTML page

</decisions>

<specifics>
## Specific Ideas

- The export service should be fully independent — no need for the frontend to be running for exports to work
- The standalone page replicates the exact visual output: Verovio SVG rendering, notehead animations, camera panning, score region cropping, background image

</specifics>
