  <action>
**1. Create `firestore.rules`** at the project root:

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // User-scoped project documents
    match /users/{userId}/projects/{projectId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }

    // Deny everything else by default
    match /{document=**} {
      allow read, write: if false;
    }
  }
}
```

Key points:
- Only authenticated users can access their own project documents
- The `request.auth.uid == userId` check ensures user-scoped access
- The catch-all rule denies access to any other document path
- These rules apply to client-side SDK access only (Admin SDK bypasses all rules)

**2. Create `storage.rules`** at the project root:

```
rules_version = '2';
service firebase.storage {
  match /b/{bucket}/o {
    // User-scoped project files
    match /users/{userId}/projects/{projectId}/{fileName} {
      // Only the file owner can read
      allow read: if request.auth != null && request.auth.uid == userId;

      // Only the file owner can write, with size constraint
      allow write: if request.auth != null
                   && request.auth.uid == userId
                   && request.resource.size < 50 * 1024 * 1024;
    }

    // Deny everything else
    match /{allPaths=**} {
      allow read, write: if false;
    }
  }
}
```

Key points:
- Only authenticated users can read/write their own files
- Write operations are limited to 50MB (matches the audio file size limit)
- The catch-all rule denies access to all other storage paths
- These are defense-in-depth -- primary enforcement is server-side session verification

These files are ready for deployment with `firebase deploy --only firestore:rules,storage` once the Firebase CLI is configured. Deployment itself is a user action (requires Firebase CLI authentication).
