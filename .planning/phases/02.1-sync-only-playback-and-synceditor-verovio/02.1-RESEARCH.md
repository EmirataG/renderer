# Phase 2.1: Sync-Only Playback & SyncEditor Verovio - Research

**Researched:** 2026-02-03
**Domain:** BPM removal, sync-only playback gating, OSMD-to-Verovio SyncEditor migration
**Confidence:** HIGH

## Summary

This phase has two distinct workstreams that are independent of each other: (1) removing BPM-based playback from RegularRenderer and App.tsx, adding sync-readiness gating to the Play button with context-aware messages; and (2) migrating SyncEditor from OSMD to Verovio using the same useVerovio hook and getEventsFromVerovio patterns established in Phases 1 and 2.

The BPM removal is straightforward code deletion with surgical UI changes. The SyncEditor migration is a focused engine swap: replace OSMD initialization with Verovio, replace OSMD Cursor-based event extraction with getEventsFromVerovio, and replace OSMD `.vf-stavenote` / `.vf-notehead` CSS selectors with Verovio's `.note` / `.notehead` / `use` selectors. All interaction patterns (click-to-select, anchor coloring, keyboard navigation, audio sync playback) remain identical in behavior.

**Primary recommendation:** Execute as two independent plans: Plan 01 for BPM removal + transport gating in RegularRenderer/App.tsx, Plan 02 for SyncEditor Verovio migration. They share no code dependencies and can be verified independently.

### Pattern 2: Transport Gating (Play Button Readiness)

**What:** Play button disabled until audio + first/last anchors are set. Context-aware message below transport.

**Readiness check logic:**
```typescript
// "First and last events" = events[0] and events[events.length - 1]
const hasAudio = !!audioUrl;
const firstEventId = events[0]?.id;
const lastEventId = events[events.length - 1]?.id;
const hasFirstAnchor = firstEventId ? syncAnchors?.has(firstEventId) : false;
const hasLastAnchor = lastEventId ? syncAnchors?.has(lastEventId) : false;
const canPlay = hasAudio && hasFirstAnchor && hasLastAnchor && events.length > 0;
```

**Context-aware message logic:**
```typescript
function getTransportMessage(hasAudio: boolean, hasFirstAnchor: boolean, hasLastAnchor: boolean): string | null {
  if (!hasAudio) return "Upload audio to enable playback";
  if (!hasFirstAnchor || !hasLastAnchor) return "Set first and last sync anchors to enable playback";
  return null; // Ready — no message
}
```

**UI location:** Inline text below the three transport buttons (Play, Pause, Reset), inside the `!isRenderMode` transport bar section (line 857).

### Pattern 3: SyncEditor Verovio Migration

**What:** Replace OSMD with Verovio in SyncEditor, preserving all interaction behavior.

**Key swap mapping:**

| OSMD Pattern | Verovio Pattern |
|-------------|-----------------|
| `new OpenSheetMusicDisplay(ref, opts)` | `useVerovio(xml, width, scale)` hook |
| `osmd.load(xml).then(() => osmd.render())` | Hook handles load+render automatically |
| `getEventsFromOsmd(osmd)` via Cursor API | `getEventsFromVerovio(toolkit, container)` |
| `target.closest('.vf-stavenote')` | `target.closest('g.note')` |
| SVG ID format: `vf-auto12345` | SVG ID format: `n1abc123` (Verovio xml:id) |
| `.vf-notehead path, .vf-notehead ellipse` | `g.notehead use` |
| `.vf-stavenote:hover .vf-notehead path` | `g.note:hover g.notehead use` |
| `osmd.Cursor` for event iteration | `toolkit.renderToTimemap()` for events |

**SyncEditor-specific style needs:**
- Notes MUST be clickable (no `pointer-events: none`)
- Hover effects on notes (cursor: pointer, brightness filter)
- Selection highlight: blue (`#3b82f6`) via inline style on `use` elements
- Anchor highlight: green (`#22c55e`) via inline style on `use` elements
- Playing highlight: orange (`#f59e0b`) via inline style on `use` elements

**SyncEditor CSS (replacing OSMD selectors):**
```css
g.note {
  cursor: pointer;
}
g.note:hover g.notehead use {
  filter: brightness(0.7);
}
```

### Pattern 4: Separate Toolkit Instances

**Recommendation: SyncEditor MUST use its own toolkit instance.**

Reasons:
1. SyncEditor and RegularRenderer render at different widths — SyncEditor is full-panel width, RegularRenderer respects scoreRegion/containerWidth
2. SyncEditor may render at different scale (no user-controlled scoreScale)
3. SyncEditor has different Verovio options (may want measure numbers visible)
4. `useVerovio` hook already creates a new toolkit per call via `createToolkit()`, which shares the WASM module singleton but gives a fresh toolkit instance

The `verovioService.ts` singleton pattern (lazy WASM module, new toolkit per `createToolkit()` call) already supports this perfectly. Each `useVerovio` call gets its own toolkit instance that shares the single WASM module.

### Recommended SyncEditor Structure After Migration

```
SyncEditor.tsx (after migration):
├── useVerovio(xml, width, scale) — renders SVG, returns toolkit
├── getEventsFromVerovio(toolkit, container) — extracts events with positions
├── Click handling: target.closest('g.note') → find event by svgId match
├── Color application: querySelector('g.notehead use') for fill/stroke
├── Anchor/selection styling via inline styles on <use> elements
├── Audio sync animation loop (unchanged logic, new selectors)
└── Keyboard navigation (unchanged)
```

### Anti-Patterns to Avoid
- **Sharing toolkit instance between views:** Different render parameters mean they need separate instances. The WASM module is shared, not the toolkit.
- **Conditional BPM/sync branching:** After BPM removal, don't leave dead code paths with `useSyncTiming` checks. Sync is the ONLY mode.
- **Keeping `useSyncTiming` variable:** After removal, all playback IS sync-based. Remove this variable entirely — the readiness check is `canPlay` (audio + anchors), not "is sync mode active."

## Common Pitfalls

### Pitfall 1: SyncEditor Width Calculation
**What goes wrong:** SyncEditor renders at 0 width because `useVerovio` requires `containerWidth > 0`.
**Why it happens:** SyncEditor is mounted with `visibility: hidden` when not active (App.tsx lines 547-555). The container may have 0 width when hidden.
**How to avoid:** The current App.tsx uses `position: absolute` with `width: 100%` when hidden, which should give non-zero width. But verify that `useVerovio` receives a valid width. May need to measure the parent container width explicitly or pass it as a prop.
**Warning signs:** SyncEditor shows blank/no SVG when first switching to sync view.

### Pitfall 2: Event ID Mismatch Between Views
**What goes wrong:** Anchors set in SyncEditor don't match events in RegularRenderer.
**Why it happens:** Both views generate events independently with `evt-0`, `evt-1`, etc. IDs match because both use `getEventsFromVerovio` which generates IDs from the same timemap order.
**How to avoid:** Both views MUST use the same event extraction function (`getEventsFromVerovio`) so IDs are consistent. The current SyncEditor uses its own `getEventsFromOsmd` which also generates `evt-N` IDs — after migration to `getEventsFromVerovio`, IDs will match because the timemap ordering is deterministic.
**Warning signs:** Setting an anchor in SyncEditor on event `evt-5` doesn't correspond to the same musical moment in RegularRenderer.

### Pitfall 3: Click Target Detection in Verovio SVG
**What goes wrong:** Clicking on a note doesn't select it because the click target doesn't match expected selectors.
**Why it happens:** Verovio SVG structure is `<g class="note"> <g class="notehead"> <use>`. The click target may be the `<use>` element, not the `<g class="note">` group. Need `target.closest('g.note')` which handles bubbling correctly.
**How to avoid:** Use `target.closest('g.note')` for click detection. The `closest()` API traverses up the DOM tree, so clicking on any child element (use, notehead group) will find the parent note group.
**Warning signs:** Clicking directly on a notehead doesn't trigger selection.

### Pitfall 4: Verovio SVG IDs vs OSMD SVG IDs
**What goes wrong:** Color application or event matching fails because ID format changed.
**Why it happens:** OSMD uses `vf-auto12345` format. Verovio uses xml:id values like `n1abc123` (no prefix). The `getEventsFromVerovio` function already returns Verovio IDs in `svgIds`. SyncEditor code that previously matched `evt.svgIds.some(id => id === noteId)` will work as-is IF `noteId` is extracted from the Verovio SVG correctly.
**How to avoid:** When extracting ID from clicked note: `const noteId = noteGroup.id` (not `noteGroup.getAttribute('id')` — though both work). Ensure the event's `svgIds` array contains the same ID format as the DOM element's `id` attribute.
**Warning signs:** `events.find(evt => evt.svgIds.some(id => id === noteId))` returns undefined for valid clicks.

### Pitfall 5: Incomplete BPM Removal
**What goes wrong:** Dead code referencing removed functions causes TypeScript errors or confusing code paths.
**Why it happens:** BPM code is interleaved throughout RegularRenderer. Easy to miss a reference.
**How to avoid:** After deletion, search for: `bpm`, `BPM`, `setupEvent`, `animateBPM`, `velocityRef`, `eventEndTimeRef`, `useSyncTiming`. All should be gone (except comments explaining the removal).
**Warning signs:** TypeScript compilation errors referencing undefined variables.

### Pitfall 6: SyncEditor Color Application on Verovio `<use>` Elements
**What goes wrong:** Setting `fill` on a `<use>` element doesn't change the visual appearance.
**Why it happens:** Verovio `<use>` elements reference `<defs>` shapes. The `fill` property on `<use>` works via CSS cascade — setting `style.fill` on the `<use>` element overrides the referenced shape's fill. This is confirmed working in RegularRenderer's noteAnimation.ts already.
**How to avoid:** Use the same pattern as noteAnimation.ts: set `fill`, `stroke`, and `color` on `<use>` elements via inline styles. The `color` property provides `currentColor` cascade fallback.
**Warning signs:** Notes don't change color when selected/anchored/playing.

### Example 2: SyncEditor Click Handler (Verovio)

```typescript
// Source: Adapted from existing SyncEditor.tsx handleScoreClick, targeting Verovio classes

const handleScoreClick = useCallback((event: React.MouseEvent<HTMLDivElement>) => {
  const target = event.target as Element;
  // Verovio wraps notes in <g class="note" id="n1abc123">
  const noteGroup = target.closest('g.note');

  if (!noteGroup) {
    selectEvent(null);
    return;
  }

  const noteId = noteGroup.id;

  // Find matching event — svgIds from getEventsFromVerovio contain Verovio xml:ids
  const clickedEvent = events.find(evt =>
    evt.svgIds.some(id => id === noteId)
  );

  if (clickedEvent) {
    selectEvent(clickedEvent.id);
  }
}, [events, selectEvent]);
```

### Example 3: SyncEditor Note Color Application (Verovio)

```typescript
// Source: Adapted from existing SyncEditor.tsx applyNoteColor, targeting Verovio structure

const applyNoteColor = (svgIds: string[], color: string) => {
  if (!containerRef.current) return;
  svgIds.forEach(svgId => {
    const noteGroup = containerRef.current?.querySelector(`#${CSS.escape(svgId)}`);
    if (!noteGroup) return;
    // Verovio noteheads use <use> elements, not <path>/<ellipse>
    const shapes = noteGroup.querySelectorAll<SVGGraphicsElement>('g.notehead use');
    shapes.forEach(shape => {
      shape.style.fill = color;
      shape.style.stroke = color;
      shape.style.color = color;
    });
  });
};

const clearNoteColor = (svgIds: string[]) => {
  if (!containerRef.current) return;
  svgIds.forEach(svgId => {
    const noteGroup = containerRef.current?.querySelector(`#${CSS.escape(svgId)}`);
    if (!noteGroup) return;
    const shapes = noteGroup.querySelectorAll<SVGGraphicsElement>('g.notehead use');
    shapes.forEach(shape => {
      shape.style.removeProperty('fill');
      shape.style.removeProperty('stroke');
      shape.style.removeProperty('color');
    });
  });
};
```

## Open Questions

1. **SyncEditor container width when hidden**
   - What we know: App.tsx mounts SyncEditor with `visibility: hidden` and `position: absolute` + `width: 100%`. The `useVerovio` hook requires `containerWidth > 0`.
   - What's unclear: Whether the hidden SyncEditor container measures non-zero width when initially mounted. The `flex-1` parent should have a computed width.
   - Recommendation: Measure the SyncEditor container width via a ref and pass it to `useVerovio`. If 0, defer rendering until visible. Alternatively, use the parent's width calculation.

2. **SyncEditor Verovio options**
   - What we know: Current OSMD config has `drawMeasureNumbers: true`, `drawTitle: false`, `drawComposer: false`.
   - What's unclear: Exact Verovio options to replicate this. Verovio uses `header: 'none'`, `footer: 'none'` (already in useVerovio). Measure numbers are on by default in Verovio.
   - Recommendation: Use same `useVerovio` options as RegularRenderer. The `header: 'none'` and `footer: 'none'` already suppress title/composer. Measure numbers display by default.

3. **Chord handling in SyncEditor click detection**
   - What we know: Verovio wraps chords in `<g class="chord">` containing multiple `<g class="note">`. The `getEventsFromVerovio` timemap's `on` array for a chord event contains multiple note IDs.
   - What's unclear: If clicking a note within a chord, `closest('g.note')` returns the individual note — but the event's `svgIds` array contains ALL notes in the chord.
   - Recommendation: The existing matching logic `evt.svgIds.some(id => id === noteId)` will correctly match because each individual note ID is in the chord event's svgIds. No special chord handling needed.
