# Phase 13.1: Unplayed Score Styling - Research

**Researched:** 2026-02-07
**Domain:** SVG styling, CSS clip-path, React state-driven rendering
**Confidence:** HIGH

## Summary

This phase implements visual differentiation between played and unplayed portions of a music score. The approach uses a hybrid strategy:

1. **Direct styling** for discrete elements (noteheads, stems, accidentals, dots) - these can switch style instantaneously when their associated musical event plays
2. **CSS clip-path** for continuous elements (staff lines, barlines, beams) - these span multiple notes and need progressive reveal based on playback X position

The key insight is that Verovio's SVG output preserves MEI element hierarchy with consistent class naming (`g.note`, `g.notehead`, `g.stem`, `g.staff`, `g.beam`, etc.), making CSS targeting straightforward. The playback position (X coordinate) is already computed per-frame via `lerp()` in the existing animation system, so the clip-path boundary can track this position precisely.

**Primary recommendation:** Implement unplayed styling as an opt-in Inspector feature using CSS custom properties for colors/opacity, direct style application for discrete elements, and an SVG `<clipPath>` with a dynamically-positioned `<rect>` for continuous elements.

### Pattern 1: Hybrid Styling (Direct + Clip-Path)

**What:** Discrete elements use direct style changes; continuous elements use clip-path reveal
**When to use:** When some elements have clear play/unplayed boundaries (notes) while others span the boundary (staff lines, beams)

**Element Classification:**

| Element Type | SVG Class | Styling Method | Reason |
|--------------|-----------|----------------|--------|
| Noteheads | `g.notehead` | Direct style | Discrete, instant transition |
| Stems | `g.stem` | Direct style | Tied to specific note |
| Accidentals | `g.accid` | Direct style | Tied to specific note |
| Dots | `g.dots` | Direct style | Tied to specific note |
| Flags | `g.flag` | Direct style | Tied to specific note |
| Staff lines | `g.staff > path` | clip-path | Continuous, horizontal span |
| Barlines | `g.barLine` | clip-path | Vertical, revealed by position |
| Beams | `g.beam polygon` | clip-path | Spans multiple notes |
| Ledger lines | `g.ledgerLines` | clip-path | Continuous, horizontal |
| Clefs | `g.clef` | Direct style or clip-path | Revealed at start |
| Key signatures | `g.keySig` | Direct style or clip-path | Revealed at start |
| Time signatures | `g.meterSig` | Direct style or clip-path | Revealed at start |

**Example Implementation:**
```typescript
// Source: MDN SVG clipPath documentation + Verovio SVG analysis
interface UnplayedStyleOptions {
  mode: 'dimmed' | 'invisible' | 'color';
  dimOpacity?: number;      // 0.3 default for 'dimmed'
  unplayedColor?: string;   // For 'color' mode
  playedColor?: string;     // Typically scoreColor
}

// Direct styling for discrete elements
function applyUnplayedStyleToNote(
  noteElement: SVGGElement,
  isPlayed: boolean,
  options: UnplayedStyleOptions
) {
  const targets = noteElement.querySelectorAll<SVGElement>(
    'g.notehead use, g.stem path, g.accid use, g.dots ellipse, g.flag use'
  );

  targets.forEach(el => {
    if (options.mode === 'invisible') {
      el.style.opacity = isPlayed ? '1' : '0';
    } else if (options.mode === 'dimmed') {
      el.style.opacity = isPlayed ? '1' : String(options.dimOpacity ?? 0.3);
    } else if (options.mode === 'color') {
      const color = isPlayed ? options.playedColor : options.unplayedColor;
      el.style.fill = color!;
      el.style.stroke = color!;
    }
  });
}

// Clip-path update for continuous elements
function updatePlaybackClipPath(
  clipPathRect: SVGRectElement,
  playbackX: number,
  totalWidth: number
) {
  // Reveal from left edge to current playback position
  clipPathRect.setAttribute('width', String(playbackX));
}
```

### Pattern 2: SVG ClipPath Definition in Defs

**What:** Define a single `<clipPath>` in SVG `<defs>` and reference it from multiple elements
**When to use:** For continuous elements that need progressive reveal

### Pattern 3: Per-Frame State Updates

**What:** Update styling during the existing animation frame loop
**When to use:** Integrate with the lerp-based camera animation for smooth clip-path updates

**Example Integration Point:**
```typescript
// In SingleLineRenderer.tsx animateSync() or setTimestamp()
// After computing currentXRef.current:

// Update clip-path for continuous elements
if (unplayedStylingEnabled && clipPathRectRef.current) {
  clipPathRectRef.current.setAttribute('width', String(currentXRef.current));
}

// Update discrete elements based on event index
if (unplayedStylingEnabled) {
  for (let i = 0; i < events.length; i++) {
    const isPlayed = i <= currentEventIndex;
    const event = events[i];
    event.svgIds.forEach(id => {
      const el = scoreRef.current?.querySelector(`#${CSS.escape(id)}`);
      if (el) applyUnplayedStyleToNote(el as SVGGElement, isPlayed, options);
    });
  }
}
```

### Anti-Patterns to Avoid

- **Per-element clipPath:** Don't create a separate `<clipPath>` for each element. Use one shared clipPath with `clipPathUnits="userSpaceOnUse"` to ensure consistent reveal position.

- **Animating clip-path via CSS transitions:** The clip-path position changes every frame based on lerp. CSS transitions would lag behind; update the attribute directly in JavaScript.

- **Querying DOM every frame for all elements:** Cache element references after Verovio renders. Only query once per SVG load, not per animation frame.

- **Using visibility: hidden:** This removes elements from the accessibility tree. Use `opacity: 0` or clip-path instead for invisible unplayed elements.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Color manipulation | Custom RGB parsing | Existing `interpolateColor` function in renderers | Already handles hex and rgb formats |
| Element targeting | Manual DOM traversal | CSS selectors with Verovio class names | Verovio guarantees consistent class structure |
| Frame-synced updates | setTimeout/setInterval | Existing requestAnimationFrame loop | Already synced to playback position |
| State persistence | localStorage directly | Zustand store with persist middleware | Consistent with existing stores |

**Key insight:** The existing animation system already computes playback X position per-frame. The clip-path simply needs to use this value.

## Common Pitfalls

### Pitfall 1: ClipPath Coordinate System Mismatch
**What goes wrong:** Clip-path doesn't align with score elements
**Why it happens:** `clipPathUnits="objectBoundingBox"` uses 0-1 normalized coordinates, while Verovio SVG uses absolute coordinates
**How to avoid:** Use `clipPathUnits="userSpaceOnUse"` to match Verovio's coordinate system
**Warning signs:** Clipping region appears at wrong scale or position

### Pitfall 2: Stacking Context Issues
**What goes wrong:** Clip-path creates new stacking context, breaking z-order
**Why it happens:** Any `clip-path` value other than `none` creates a stacking context (like `opacity: 0.999`)
**How to avoid:** Apply clip-path at the appropriate level in the DOM hierarchy; test z-order with borders and overlays
**Warning signs:** Score borders appear behind or in front of unexpected elements

### Pitfall 3: Performance Degradation with Many Elements
**What goes wrong:** Frame rate drops when styling hundreds of notes
**Why it happens:** Querying and mutating styles for every element every frame
**How to avoid:**
- Cache element references after SVG load
- Only update elements that cross the played/unplayed boundary
- Track last-styled event index and only style new events
**Warning signs:** Choppy playback, high CPU usage during animation

### Pitfall 4: Beams Partially Clipped Incorrectly
**What goes wrong:** Beam appears cut off mid-note instead of at playback position
**Why it happens:** Beam spans from first to last note in group; clip-path reveals based on X position
**How to avoid:** This is actually correct behavior - the beam is progressively revealed. Document this as expected.
**Warning signs:** None - this is the intended progressive reveal effect

### Pitfall 5: Inspector Settings Not Persisting
**What goes wrong:** User has to re-enable unplayed styling on each session
**Why it happens:** State stored in component local state instead of persistent store
**How to avoid:** Use Zustand with persist middleware for all Inspector settings
**Warning signs:** Settings reset on page refresh

### Pitfall 6: RegularRenderer Y-axis Confusion
**What goes wrong:** Attempting to use X-based clip-path for vertical scrolling renderer
**Why it happens:** RegularRenderer scrolls vertically (Y-axis), not horizontally
**How to avoid:**
- For RegularRenderer: Use system-based approach - style all elements in systems above current as "played"
- Or implement Y-based clip-path that reveals from top
**Warning signs:** Horizontal clipping in a vertical layout

- `clip-path: url()` with objectBoundingBox: Works but userSpaceOnUse is simpler for absolute positioning

## Open Questions

Things that need validation during implementation:

1. **Beam/Tie Clipping Aesthetics**
   - What we know: Beams will be progressively revealed left-to-right
   - What's unclear: Whether this looks good or if beams should only appear once fully revealed
   - Recommendation: Implement progressive reveal first, add option for "reveal complete" if needed

2. **RegularRenderer Strategy**
   - What we know: RegularRenderer scrolls vertically, not horizontally
   - What's unclear: Whether to use Y-based clip-path or system-based styling
   - Recommendation: Start with system-based approach (style whole systems as played/unplayed), simpler than Y-axis clipping

3. **Initial Render State**
   - What we know: Before playback starts, entire score is "unplayed"
   - What's unclear: Should the score start fully visible or in unplayed style?
   - Recommendation: Start in unplayed style to show the feature is active; provide "preview" toggle option
